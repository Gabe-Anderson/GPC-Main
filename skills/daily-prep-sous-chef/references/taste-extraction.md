# Taste Extraction

The core product taste: this is not a daily brief. A brief summarizes the user's life back to them. A sous-chef prepares the ingredients, handles safe work, and surfaces only the real decision.

## Transferable Taste

- Execution over briefing.
- Harvard-MBA-caliber chief-of-staff posture: commercially sharp, personally aware, context-rich, and forward-looking.
- Cross-surface synthesis before output.
- Structured extraction ledger before synthesis.
- Due-today/recently updated coverage invariant: every live task gets a single state, sources checked, and next safe move.
- Link and pointer follow-through before blocking: task links, thread IDs, docs, repo paths, named projects, and fallback title/subject/sender searches.
- Repo/source locator pass for code and project work before asking the user where the source lives.
- Big Thing Radar to rank risks and opportunities.
- Specialist passes that return typed findings before final synthesis.
- Tasks as inputs/status surfaces, not outputs.
- Traffic-light autonomy.
- Low-noise communication.
- Priority-aware filtering.
- Next-72-hour logistics preflight.
- Sent-mail/action reconciliation.
- Stale-loop suppression: completed tasks, sent replies, and resolved threads override old run state.
- Priority relationship prep with recent notes/transcripts and public signals where useful.
- Regression checks, a fresh-task adversarial audit, and a final red-team pass against known miss patterns.
- Parent-task integrity.
- Allocation / Removal / Tiny Hinge as silent internal lenses.

## Do Not Transfer

- Private names.
- Personal docs.
- Account names.
- File paths.
- Task IDs.
- Relationship details.
- Dating context.
- Financial specifics.
- Another user's exact operating system.

## Test For Quality

Ask:

- Did it remove work before the user asked?
- Did it surface fewer, better decisions?
- Did it understand what actually matters?
- Did it extract the key fields before claiming a surface was checked?
- Did every due-today or recently updated task receive a ledger state and next safe move?
- Did it follow task links, stale email links, doc links, repo/source pointers, and named systems before calling something blocked?
- Did it catch missing logistics, unconfirmed important attendees, stale task state, and priority relationship prep?
- Did it suppress stale loops when the task/thread was already resolved?
- Did it run a critic pass against known miss patterns?
- Did it ask what the user may have deliberately placed to test the workflow?
- Did it avoid creating more task/admin debt?
- What did it prepare that a normal daily brief would have missed?
